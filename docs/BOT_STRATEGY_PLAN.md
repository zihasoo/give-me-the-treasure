# 봇 전략 현황과 로드맵

봇 의사결정 구조의 현재 상태와 앞으로 할 일을 정리한 살아있는 문서다.
**이 문서만 보고 다음 작업을 코드로 옮길 수 있도록** 진단·구현 위치·알고리즘·검증을 함께 적는다.

## 0. 한 줄 요약 (지금 읽을 사람에게)

**S7이 현재 최신·기본 봇이다.** Heuristic → S1 → … → S6 → S7 순으로 진화했다. 직전 세대 S6는 A/B 비교용 베이스라인으로만 남겼고(대기실에서 선택 가능), 그 이전은 제거했다. S7은 사람 vs S6 로그(2026-06-16, 사람 32:20 승)가 짚은 세 약점 — ①선택 오판으로 보물(와일드) 헌납, ②저주를 부채로 안 봄, ③3장 세트 난사 — 을 고친 버전이다. **강함의 척도는 봇끼리 승률이 아니라 사람 vs 봇 플레이 로그 분석이다.** → [§4 다음 작업](#4-다음-작업-로드맵).

## 1. 현재 구조

봇은 `BotStrategy` 인터페이스로 모든 의사결정을 위임한다(전략 패턴).

- `decideSplit(SplitContext)` / `decideChoice(ChoiceContext)` / `decideHelpers` / `planCashIn(CashInContext, opponentCoins)` / `decideTeamDistribution`

구현체와 유틸 (`com.oop.payday.bot`):

| 클래스 | 역할 |
|---|---|
| `S7BotStrategy` | 현재 최신·기본 봇. S6에 ①적대적 뒷면·보물 평가(`hiddenEv`/`wildSwing`) ②저주 부채(`CURSE_LIABILITY`) ③같은색 연속 보류 강화를 더함. |
| `S6BotStrategy` | 직전 세대. A/B 비교용 베이스라인으로 동결 보존. |
| `BotCardEvaluator` | 카드 잠재력 점수(`potentialScore`)·세트 코인(`bestCashCoin`)·버릴 카드(`chooseDiscard`). |
| `CashInPlanOptimizer` | 환금 단계 조합 탐색. `plan`/`pickSetsToHold`·다중 세트 보류. 세대별 정책은 `Tuning`(S6=동결, S7=저주 보유공간 제외·TUSKER 가드·보류 임계 완화)으로 갈아끼움. |
| `BotKind` | 대기실 봇 슬롯 선택지(`S7`·`S6` 노출, 향후 난이도별 확장 예정). |
| `PlayLogWriter` | 오프라인 게임 진행을 `logs/play-*.log`에 사람 가독 형식으로 기록. 사람 vs 봇 사후 분석용. |

**컨텍스트 분리 원칙:** 봇 전용·비동기화 정보(`opponentCoins`)는 `beginCashIn`의 별도 인자로 흘린다. 공개 정보(`opponentHoldings`)는 `CashInContext`/`CashInContextDto`에 포함해 네트워크로 전달한다.

**검증:** `HeadlessBotGameTest`(`botsCanFinishPracticeGameWithoutUi` — 봇 대전이 무효 행동 없이 승자를 내고 끝나는 회귀 안전망 + `playLogCapturesFullGame` — 로그 형식 회귀) + `HelperActionModelTest`(S6 환금 동작 회귀). 봇끼리 승률 A/B는 더 이상 측정하지 않으며, 강함의 척도는 사람 vs 봇 로그다.

## 2. 진화 기록 (Heuristic → S6)

각 세대의 핵심 변화와 봇끼리 대전 결과만 남긴다. 구현체는 제거됐으나 설계 맥락 보존을 위해 요약한다.

| 버전 | 핵심 추가 | 이전 대비 승률 |
|---|---|---|
| Heuristic | 규칙 기반. 세트 코인 가치만 보고 단순 선택. 비교 하한선. | — |
| S1 | 점수 기반 기준선. 환금 조합 최적화 + 카드 잠재력 평가. | — |
| S2 | S1 + 분할 블러핑 명시화(`LURE_WEIGHT`) + 자기 팀 임박 환금. | S1 대비 53% |
| S3 | S2 + 상황 컨텍스트(보유 카드·양 팀 코인). 시너지·적응형 선택자 모델·승리 확정 분할. | S2 대비 54% |
| S4 | S3 + **환금 보류**(`planWithHold`) + **상대 holdings 견제**(`oppSynergy`). 전략 축 변경. | S3 대비 **79%** |
| S5 | S4 + 보류 판단 정밀화(이진→연속 기대값, 상대 손패 반영). `CashInContext`에 `opponentHoldings` 추가. | S4 대비 56% |
| S6 | S5 + 현실적 선택자 모델·상시 maximin·다중 세트 보류·뒷면 저주 떠넘김. + `PlayLogWriter`. | S5 대비 **72%** |
| S7 | S6 + ①적대적 뒷면·보물 평가 ②저주 부채 ③같은색 연속 보류 강화. 사람 vs S6 로그 기반 교정. | (봇 A/B 미측정) |

**해석:** 파라미터 미세조정(S1→S3) 단계는 3~4%p 개선에 머물렀고, 전략 축을 바꾼 S4가 +29%p 도약이었다. S6의 높은 봇 승률도 사람 상대 강함을 보증하지 않았다 — 1차(2026-06-15, user 31:22)·2차(2026-06-16, user 32:20) 플레이 모두 봇이 졌다. S7부터는 봇끼리 승률을 측정하지 않고 사람 vs 봇 로그로만 검증한다.

## 3. S6 상세 — 무엇을, 왜 바꿨나

#### 분할 호구 + 선택 오판 교정

S5는 `chooserTakesA = visA > visB`(공개 코인 최대)로 상대 선택을 단순 가정했고, maximin 안전장치는 종반에만 켜졌다. S6는 `predictedChooserValue`(즉시 코인 + 상대 손패 시너지 + 뒷면 기대값)로 현실적 선택자 모델을 쓰고, `BASE_MAXIMIN_WEIGHT(=50)`으로 maximin을 상시 켠다. → 중반에도 무방비 분할이 줄고, 상대 색연속 완성 묶음을 순진하게 내주지 않는다.

#### 저주 떠넘김 — 뒷면 숨김만

보이는 저주는 생각하는 상대에게 못 넘긴다(그냥 다른 묶음을 집음). "보이는 저주를 상대 묶음에 몰아넣기" 라우팅은 제거했다. 남긴 것은 `hiddenCurseDumpScore` — 뒷면 카드가 저주이고 선택자가 가져갈 것으로 예측된 묶음이면 보너스(`CURSE_HIDDEN_DUMP`).

#### 다중 세트 보류 (`pickSetsToHoldS6`)

S5는 한 세트만 보류해 소형 세트를 자잘하게 냈다(1차 플레이 패인). S6는 성장 기대값(`HOLD_GROWTH_THRESHOLD`)이 큰 세트를 한도가 허락하는 한 여러 개 보류해 크게 키운다.

#### 플레이 로그 (`PlayLogWriter`)

오프라인 게임을 `logs/play-*.log`에 분할자 손패·양 팀 보관·코인·제시 묶음·분배 결과·환금 세트를 사람 가독 형식으로 기록. 기본 ON(`-Dpayday.playlog=false`로 끔).

## 3.5 S7 상세 — 2차 로그(2026-06-16)가 짚은 세 약점 교정

2차 플레이(사람 32:20 승)는 R4까지 0~2코인 접전이었으나, 사람이 R5(9코인)·R6(15코인)에서 **굉장한 보물(와일드)** 을 재활용해 대형 세트를 뽑으며 끝났다. 그 와일드는 **R3에서 봇이 헌납**한 것이다. 봇은 한 번도 4장 이상 세트를 못 냈고(2·2·4·4·6·6), 사람은 4·5장(9·15코인)을 냈다. 세 약점이 맞물려 있었다.

#### ① 선택 오판 — 보물 헌납 (`hiddenEv`·`wildValue`·물량 기반 `predictedChooserValue`, `S7BotStrategy`)

S6 `decideChoice`는 뒷면을 고정 보너스 20으로만 봐서, 사람이 *얇은 묶음 뒤에 보물을 숨기고 두꺼운 미끼 묶음을 내미는* 분할(R3: 묶음0=청록6+뒷면[보물] vs 묶음1=3장)에 걸려 미끼 쪽을 집었다. S7은 `hiddenEv`로 **적대적 뒷면 추론**을 한다 — 이 묶음이 상대 묶음보다 공개상 빈약할수록(코인↓·장수↓) 분할자가 값진 카드를 숨겨 날 다른 데로 유도한 것이라 보고 뒷면 기대값을 키운다(종반 가중). 같은 모델을 `predictedChooserValue`에도 적용해 봇이 값진 카드를 얇은 묶음에 숨겨 흘리던 누수(R4·R8)도 줄인다.

**와일드 보호(`wildValue=WILD_GRAB`):** 와일드(굉장한 보물)는 최대 15코인 세트를 완성하는 핵심 자원이라 `myKeepValue`·`standaloneValue`·`predictedChooserValue`에 전부 반영한다. 선택 시 보이는 와일드는 무조건 확보(+)·견제(−)하고, **분할 시**엔 와일드를 공개로 두면 선택자가 무조건 가져간다고 예측하므로 와일드를 뒷면으로 숨겨 지키도록 유도한다(사람 vs S7 로그 171541-R6에서 봇이 와일드를 공개로 내줘 사람이 10코인 세트를 만든 누수 교정).

**이유 없는 1:4 분할 방지(`predictedChooserValue` 물량 기반, `CARD_MATERIAL`):** 초기 S7은 `hiddenEv`(얇을수록 값진 뒷면)를 분할 예측에도 써서 "사람이 빈 묶음(뒷면 1장)을 가져갈 것"이라 착각 → 1:4로 나눠 4장을 헌납했다(172915 R1·R5: 뒷면 1에 저주/평범 카드만 두고 4장을 줌). 교정: **선택자 예측은 카드 물량(장수)에 비례**시킨다 — 4장 묶음이 1장(+뒷면) 묶음보다 매력적이고, 뒷면은 카드 1장 + 약간의 미지 프리미엄일 뿐이다. 이러면 봇은 1:4가 *자기에게 1장만 남기는* 손해임을 알아 2:3으로 균형 분할하고, 1:4는 **남기는 1장이 와일드 등 고가치일 때만** 둔다(`hiddenEv`는 봇 자신의 선택 `decideChoice`에만 남겨 숨은 와일드를 노린다).

#### ② 저주 운용 (`CURSE_LIABILITY` + `Tuning.S7`)

**저주 비용 구조(규칙 §3-1 + `Game.applyDiscard`):** 저주를 없애는 길은 셋뿐 — (a) 매칭 숫자 보물을 포함한 세트에 끼워 **무료 처분**(0코인), (b) **자발적 버림 2코인**, (c) **척후 바이퍼**(전부 무료 처분 + 처분 장수만큼 코인 획득). 그냥 버려도 2코인이 든다는 게 핵심이다.

S6의 약점은 **저주를 부채로 안 본 것**이다 — keep/choice 평가에서 0(중립)으로 봐 저주 묶음을 태연히 떠안았고(R2·R4), 저주가 보유 한도를 잡아먹어 키울 세트를 즉시 환금하게 됐다(R7: 보유 한도 7을 저주 3장이 점유). S7 교정:

- **분할/선택에 부채 반영(`CURSE_LIABILITY`):** 애초에 저주를 안 떠안도록 `myKeepValue`에 저주당 음의 가치를 깐다.
- **보류 공간 확보(`curseFreeHoldRoom`):** 저주는 처분 예정이므로 보유 한도 계산에서 제외해, 저주가 키울 세트를 막지 않게 한다.
- **무료 처분 보너스는 한도 초과일 때만(`freeCurseRewardOnlyOverLimit`):** 한도 여유가 있을 땐 무료 처분은 이득이 아니라 *덜 나쁜 손해*다(저주가 보물이었다면 더 크게 냈을 것) → 보너스 0, 저주 털려고 세트를 일찍 내는 왜곡 방지. **한도가 넘치면** 무료 처분이 2코인 강제 버림을 피하는 실질 이득이라 보너스(저주 1장당 ≈2코인)를 켠다.
- **바이퍼는 한도 초과 구제용으로만(`viperOnlyWhenOverLimit`):** 한도 여유가 있으면 매칭 세트로 무료 처분하거나 더 모았다가 쓰는 게 나아 1회용 바이퍼를 아낀다. 환금 뒤에도 한도가 넘칠 때만 발동(높은 점수 환금으로 해결되면 환금 우선 — 환금이 먼저 실행되므로 자연 성립).
- **TUSKER 는 건드리지 않음:** 그냥 버려도 2코인이 드니, TUSKER 로 한도를 풀어 다음 라운드 무료 처분을 노리는 건 합리적이다(R3의 TUSKER 사용은 자해가 아니었다 — 초기 분석 정정).

#### ③ 3장 세트 난사 (`Tuning.S7`)

같은색 연속은 천장이 가장 높은데(3장=6 → 4장=9 → 5장=15) S6는 3장에서 즉시 환금했다. 같은색 연속 성장은 *특정 색의 특정 숫자 한 장*이 필요해 성장 기대값이 낮고 임계(0.8)를 못 넘긴 탓. S7은 보류 임계를 낮추고(`0.5`) 같은색 연속 성장에 가중(`flushGrowthMultiplier`)을 둔다. ②로 보유 한도가 트여야 실제로 보류되므로 ②·③은 함께 작동한다.

> **주의:** 위 가중치(`BASE_HIDDEN_EV=60`, `COUNT_LURE=60`, `HIDDEN_SUSPICION=120`, `WILD_GRAB=600`, `CURSE_LIABILITY=120`, 보류 임계 0.5)는 로그 1판 기반 1차 추정값이다. 다음 사람 vs S7 로그로 재조정 대상.

- **171541 (와일드 보호·1:4 수정 전, 사람 32:31, 거의 짐):** S7이 R4까지 15:0 압도. 패인 = R6에서 봇이 **와일드를 공개로 분할해 헌납**(→ 사람 10코인 세트) + R8 완성 플러시 헌납. → **와일드 보호** 교정(§3.5 ①).
- **172915 (저주·VIPER 수정 후, 사람 38:15, 운):** 봇이 R2에서 **숨은 와일드를 간파해 확보**, R4에서 **5장 연속 10코인 + 저주 2장 무료 처분**(over-limit) — 의도대로 작동. 패인은 사람이 JUNK_DEALER+슬쩍하기로 **버려진 와일드를 3번 재활용**(19코인) — 통제 불가한 상대 도우미 드래프트 운. VIPER도 한도 여유엔 안 터졌다.
- **174633 (1:4 수정 직전, 사람 33:15):** 봇이 R5에 **5장 연속 10코인** + 클린한 저주 처분 등 똑똑해짐. 패인 = **1:4 분할 남발(R2·R6)** — 빈 묶음+뒷면(저주)을 두고 4장을 헌납, 그 8장이 사람의 빨강·파랑 플러시(15코인)가 됐다. → **1:4 수정**(물량 기반 `predictedChooserValue`)의 직접 증거. 이 수정 반영 버전에서 재확인 필요.
- **175208 (1:4 수정 후, 사람 30:23):** 봇이 R1~R4까지 6→10→12코인으로 앞섰으나 R6에서 사람이 **빨강 연속+같은색 4장 9코인 + 음속의 쿠쿠 3코인**(한 라운드 12코인)으로 역전해 그대로 끝났다. 도우미: 사람=척후 바이퍼·음속의 쿠쿠 / 봇=경찰견 알파·완력의 투스커. **와일드 뒷면 전략 두 번 모두 실패:** R2에서 봇이 굉장한 보물을 뒷면으로 숨겨 `노랑3+뒷면[와일드]` vs `빨강1+청록3+청록6(3장)` 구도를 만들었으나 사람이 얇은 묶음에 와일드가 있다고 간파해 선택 → 사람이 R4에서 와일드를 연속4장 6코인 세트에 사용. R8에서 봇이 또 굉장한 보물을 뒷면으로 숨겨 `슬쩍하기+노랑3` vs `노랑4+저주+뒷면[와일드]` 분할 → 사람이 또 간파해 와일드 획득, 같은숫자3장 5코인으로 30코인 달성. **결론: 1장+뒷면 vs 3장 이상 구도에서 숙련 플레이어는 뒷면에 와일드가 있다고 읽는다** — 와일드 보호를 뒷면 숨기기에만 의존하는 전략이 통하지 않음. **저주 선택(R5):** 봇이 선택자로서 묶음0(청록1·저주받은그림4) vs 묶음1(슬쩍하기·저주받은그림3+뒷면) 중 저주가 공개인 묶음0을 선택해 저주4를 R8까지 보관에 쥐고 결국 미처분 — CURSE_LIABILITY가 선택 단계에서 충분히 작동하지 않은 것인지 확인 필요. **보류 실패:** 봇 최대 세트 3장(4코인)으로 174633 이전과 동일한 한계 — 사람이 빨강 연속+같은색을 R6에서 폭발시켰는데 봇이 상대 빨강 시너지 누적을 견제하지 못했다.

다음 사람 vs S7 로그에서 확인할 것:

- 와일드 보호가 작동하는가 — 봇이 와일드를 손패에 쥐었을 때 공개로 안 내주고 뒷면으로 지키는가?
- 보류가 실제로 4·5장 세트로 이어지는가, 아니면 키우지 못하고 보유만 하다 흘리는가?
- 봇이 와일드를 환금에 써버린 뒤 상대가 회수·재활용하는 패턴 — 봇 드래프트에 JUNK_DEALER 가중 ↑ 또는 와일드를 더 오래 쥐고 키우는 보류가 필요한가? (상대 도우미 비공개라 §4.3 영역과 겹침)

### 4.2 [그다음] 난이도별 봇

`BotKind`에 `EASY`/`NORMAL`/`HARD` 추가. EASY = 강한 결정에 확률적 무작위를 섞는 래퍼 전략(일부러 약하게). NORMAL = S7의 가중치 낮춤, HARD = S7 풀파워.

### 4.3 [장기] 매우 어려움

상대 공개 카드 누적 추적(버림 더미·역할별 장기 전략), 시뮬레이션 기반 선택(상대 응답 분포를 가정한 lookahead). 범위가 가장 크고 4.1 분석 결과가 선행되어야 한다.

## 5. 설계 원칙 (유지)

- 새 전략은 `BotStrategy` 인터페이스를 유지한다.
- 게임 엔진 규칙 로직과 봇 평가 로직을 분리한다(`SetEvaluator`/`CashInEvaluator`).
- 봇 전용 컨텍스트는 직렬화 경로 밖에 둔다 — 네트워크/UI 미러(DTO)를 오염시키지 않는다.
- 봇에 넘기는 정보는 규칙상 공개된 것만(보관 카드 O, 상대 도우미·손패 X).
- 각 액션 생성 전에 검증해 무효 행동을 내지 않는다(현재 0건 — 회귀 시 테스트가 잡는다).
