# 봇 전략 현황과 로드맵

봇 의사결정 구조의 현재 상태와 앞으로 할 일을 정리한 살아있는 문서다.
**이 문서만 보고 다음 작업을 코드로 옮길 수 있도록** 진단·구현 위치·알고리즘·검증을 함께 적는다.

---

## 0. 한 줄 요약 (지금 읽을 사람에게)

**S6가 현재 유일한 봇 구현체다.** Heuristic → S1 → S2 → S3 → S4 → S5 → S6 순으로 진화했으며, 이전 구현체는 역할을 다해 제거했다. S6는 현실적 선택자 모델·상시 maximin·다중 세트 보류·뒷면 저주 떠넘김을 갖춘 버전이다. **강함의 척도는 봇끼리 승률이 아니라 사람 vs 봇 플레이 로그 분석이다.** → [§4 다음 작업](#4-다음-작업-로드맵).

---

## 1. 현재 구조

봇은 `BotStrategy` 인터페이스로 모든 의사결정을 위임한다(전략 패턴).

- `decideSplit(SplitContext)` / `decideChoice(ChoiceContext)` / `decideHelpers` / `planCashIn(CashInContext, opponentCoins)` / `decideTeamDistribution`

구현체와 유틸 (`com.oop.payday.bot`):

| 클래스 | 역할 |
|---|---|
| `S6BotStrategy` | 현재 최신·유일 봇 구현체. 현실적 선택자 모델·상시 maximin·다중 세트 보류·뒷면 저주 떠넘김. |
| `BotCardEvaluator` | 카드 잠재력 점수(`potentialScore`)·세트 코인(`bestCashCoin`)·버릴 카드(`chooseDiscard`). |
| `CashInPlanOptimizer` | 환금 단계 조합 탐색. `planWithHoldS6`(다중 보류·도우미 강화)·`pickSetsToHoldS6` 포함. |
| `BotKind` | 대기실 봇 슬롯 선택지(`S6` 노출, 향후 난이도별 확장 예정). |
| `PlayLogWriter` | 오프라인 게임 진행을 `logs/play-*.log`에 사람 가독 형식으로 기록. 사람 vs 봇 사후 분석용. |

**컨텍스트 분리 원칙:** 봇 전용·비동기화 정보(`opponentCoins`)는 `beginCashIn`의 별도 인자로 흘린다. 공개 정보(`opponentHoldings`)는 `CashInContext`/`CashInContextDto`에 포함해 네트워크로 전달한다.

**검증:** `HeadlessBotGameTest`의 `S5vsS6SeedReport`(회귀 안전망: S6 승률 ~72.5%, 무효 환금 0) + `playLogCapturesFullGame`(로그 형식 회귀). 봇 vs 봇 승률은 회귀 방지용이며, 강함의 척도가 아니다.

## 2. 진화 기록 (Heuristic → S6)

각 세대의 핵심 변화와 봇끼리 대전 결과만 남긴다. 구현체는 제거됐으나 설계 맥락 보존을 위해 요약한다.

| 버전 | 핵심 추가 | 이전 대비 승률 |
|---|---|---|
| Heuristic | 규칙 기반. 세트 코인 가치만 보고 단순 선택. 비교 하한선. | — |
| S1 | 점수 기반 기준선. 환금 조합 최적화 + 카드 잠재력 평가. | — |
| S2 | S1 + 분할 블러핑 명시화(`LURE_WEIGHT`) + 자기 팀 임박 환금. | S1 대비 53% |
| S3 | S2 + 상황 컨텍스트(보유 카드·양 팀 코인). 시너지·적응형 선택자 모델·승리 확정 분할. | S2 대비 54% |
| S4 | S3 + **환금 보류**(`planWithHold`) + **상대 holdings 견제**(`oppSynergy`). 전략 축 변경. | S3 대비 **79%** |
| S5 | S4 + 보류 판단 정밀화(이진→연속 기대값, 상대 손패 반영). `CashInContext`에 `opponentHoldings` 추가. | S4 대비 56% |
| S6 | S5 + 현실적 선택자 모델·상시 maximin·다중 세트 보류·뒷면 저주 떠넘김. + `PlayLogWriter`. | S5 대비 **72%** |

**해석:** 파라미터 미세조정(S1→S3) 단계는 3~4%p 개선에 머물렀고, 전략 축을 바꾼 S4가 +29%p 도약이었다. S6의 높은 봇 승률도 사람 상대 강함을 보증하지 않는다 — 1차 플레이(2026-06-15, user 31:22 승)에서 봇은 2~5코인 소형 세트를 반복해서 냈고, 사람은 연속 5장(10코인)+4장(6코인)을 지었다.

## 3. S6 상세 — 무엇을, 왜 바꿨나

#### 분할 호구 + 선택 오판 교정

S5는 `chooserTakesA = visA > visB`(공개 코인 최대)로 상대 선택을 단순 가정했고, maximin 안전장치는 종반에만 켜졌다. S6는 `predictedChooserValue`(즉시 코인 + 상대 손패 시너지 + 뒷면 기대값)로 현실적 선택자 모델을 쓰고, `BASE_MAXIMIN_WEIGHT(=50)`으로 maximin을 상시 켠다. → 중반에도 무방비 분할이 줄고, 상대 색연속 완성 묶음을 순진하게 내주지 않는다.

#### 저주 떠넘김 — 뒷면 숨김만

보이는 저주는 생각하는 상대에게 못 넘긴다(그냥 다른 묶음을 집음). "보이는 저주를 상대 묶음에 몰아넣기" 라우팅은 제거했다. 남긴 것은 `hiddenCurseDumpScore` — 뒷면 카드가 저주이고 선택자가 가져갈 것으로 예측된 묶음이면 보너스(`CURSE_HIDDEN_DUMP`).

#### 다중 세트 보류 (`pickSetsToHoldS6`)

S5는 한 세트만 보류해 소형 세트를 자잘하게 냈다(1차 플레이 패인). S6는 성장 기대값(`HOLD_GROWTH_THRESHOLD`)이 큰 세트를 한도가 허락하는 한 여러 개 보류해 크게 키운다.

#### 플레이 로그 (`PlayLogWriter`)

오프라인 게임을 `logs/play-*.log`에 분할자 손패·양 팀 보관·코인·제시 묶음·분배 결과·환금 세트를 사람 가독 형식으로 기록. 기본 ON(`-Dpayday.playlog=false`로 끔).

## 4. 다음 작업 (로드맵)

> **측정 축은 사람 vs 봇 로그 분석이다.** 봇 vs 봇 승률은 회귀 안전망으로만 쓴다.

### 4.1 [진행 중] 사람 vs S6 로그 분석 → 다음 약점 파악

1차 플레이(2026-06-15) 패인: 봇이 소형 세트를 자잘하게 환금(다중 세트 보류로 교정). 2차 이후 로그에서 확인해야 할 것:

- 환금 세트 크기 분포 — 사람보다 여전히 작은가?
- 분할 무방비 — 사람이 매판 상대 묶음을 쉽게 골라가는가?
- 저주·고가치 보물 운용 — 봇이 의도적으로 활용하는가, 무작위처럼 보이는가?

로그에서 이 패턴이 반복되면 해당 항목을 다음 튜닝 대상으로 삼는다.

### 4.2 [다음 우선] 난이도별 봇

`BotKind`에 `EASY`/`NORMAL`/`HARD` 추가. EASY = 강한 결정에 확률적 무작위를 섞는 래퍼 전략(일부러 약하게). NORMAL = S6의 가중치 낮춤, HARD = S6 풀파워.

### 4.3 [장기] 매우 어려움

상대 공개 카드 누적 추적(버림 더미·역할별 장기 전략), 시뮬레이션 기반 선택(상대 응답 분포를 가정한 lookahead). 범위가 가장 크고 4.1 분석 결과가 선행되어야 한다.

## 5. 설계 원칙 (유지)

- 새 전략은 `BotStrategy` 인터페이스를 유지한다.
- 게임 엔진 규칙 로직과 봇 평가 로직을 분리한다(`SetEvaluator`/`CashInEvaluator`).
- 봇 전용 컨텍스트는 직렬화 경로 밖에 둔다 — 네트워크/UI 미러(DTO)를 오염시키지 않는다.
- 봇에 넘기는 정보는 규칙상 공개된 것만(보관 카드 O, 상대 도우미·손패 X).
- 각 액션 생성 전에 검증해 무효 행동을 내지 않는다(현재 0건 — 회귀 시 테스트가 잡는다).
