# 봇 전략 현황과 로드맵

봇 의사결정 구조의 현재 상태와 앞으로 할 일을 정리한 살아있는 문서다.
**이 문서만 보고 다음 작업을 코드로 옮길 수 있도록** 진단·구현 위치·알고리즘·검증 기준을 함께 적는다.

---

## 0. 한 줄 요약

**S7이 현재 최신·기본 봇이다.** Heuristic → S1 → ... → S6 → S7 순으로 진화했다. S7은 사람 vs S6 로그(2026-06-16, 사람 32:20 승)가 드러낸 세 약점 — ①선택 오판으로 와일드 헌납, ②저주를 부채로 보지 않음, ③3장 세트 난사 — 을 교정했고, 이후 사람 vs S7 3판(2026-06-17, 봇 1승 2패)에서 종반 결승 라운드 선택 평탄화 버그를 발견해 `SECURE_MARGIN_UNIT`으로 고쳤다.

다음 단계는 S7 가중치 미세조정보다 **S8급 구조 개선**이 우선이다. 핵심은 `공개 정보 메모리` + `상대 선택 확률 모델` + `실제 환금 결과 평가` + `2v2 분배/도우미 상황화`다.

---

## 1. 현재 구조

봇은 `BotStrategy` 인터페이스로 모든 의사결정을 위임한다(전략 패턴).

- `decideSplit(SplitContext)`
- `decideChoice(ChoiceContext)`
- `decideHelpers(List<HelperCard>, chooseCount)`
- `planCashIn(CashInContext, opponentCoins)`
- `decideTeamDistribution(acquired, memberHoldings)`

구현체와 유틸 (`com.oop.payday.bot`):

| 클래스 | 역할 |
|---|---|
| `S7BotStrategy` | 현재 최신·기본 봇. S6에 적대적 뒷면 추론, 와일드 보호, 저주 부채, 같은색 연속 보류 강화, 종반 실현 코인 마진을 더함. |
| `S6BotStrategy` | 직전 세대. 비교/회귀용 베이스라인으로 동결 보존. |
| `BotCardEvaluator` | 카드 잠재력 점수(`potentialScore`), 즉시 세트 코인(`bestCashCoin`), 버릴 카드(`chooseDiscard`) 평가. |
| `CashInPlanOptimizer` | 환금 조합 탐색, 다중 세트 보류, 도우미 사용, 강제 처분 계획. 세대별 정책은 `Tuning`으로 갈아끼움. |
| `BotKind` | 대기실 봇 슬롯 선택지. 현재 `S7`/`S6` 노출, 향후 난이도 확장 예정. |
| `PlayLogWriter` | 오프라인 게임 진행을 `logs/play-*.log`에 기록. 사람 vs 봇 사후 분석용. |

**컨텍스트 분리 원칙:** 봇 전용 정보(`opponentCoins`)는 `beginCashIn`의 별도 인자로 흐른다. 규칙상 공개 정보(`opponentHoldings`)는 `CashInContext`/`CashInContextDto`에 포함해 네트워크로 전달한다.

**현재 한계:** S7은 한 라운드 안의 분할/선택/환금은 꽤 정교하지만, 장기 상태 추적·상대 성향 추정·2v2 팀 분배·정식 룰 보너스까지 포함한 실현 가치 평가는 아직 얇다.

## 2. 진화 기록

각 세대의 핵심 변화와 당시 봇끼리 대전 결과를 남긴다. 구현체가 제거된 세대도 설계 맥락 보존을 위해 요약한다.

| 버전 | 핵심 추가 | 이전 대비 승률 |
|---|---|---|
| Heuristic | 규칙 기반. 세트 코인 가치만 보고 단순 선택. 비교 하한선. | - |
| S1 | 점수 기반 기준선. 환금 조합 최적화 + 카드 잠재력 평가. | - |
| S2 | S1 + 분할 블러핑 명시화(`LURE_WEIGHT`) + 자기 팀 임박 환금. | S1 대비 53% |
| S3 | S2 + 상황 컨텍스트(보유 카드·양 팀 코인). 시너지·적응형 선택자 모델·승리 확정 분할. | S2 대비 54% |
| S4 | S3 + 환금 보류(`planWithHold`) + 상대 보관 카드 견제(`oppSynergy`). 전략 축 변경. | S3 대비 **79%** |
| S5 | S4 + 보류 판단 정밀화(이진→연속 기대값, 상대 손패 반영). `CashInContext`에 `opponentHoldings` 추가. | S4 대비 56% |
| S6 | S5 + 현실적 선택자 모델·상시 maximin·다중 세트 보류·뒷면 저주 떠넘김 + `PlayLogWriter`. | S5 대비 **72%** |
| S7 | S6 + 적대적 뒷면/와일드 평가, 저주 부채, 같은색 연속 보류 강화, 종반 실현 코인 마진. 사람 vs S6/S7 로그 기반 교정. | S6 대비 58.2% |

**해석:** S1→S3의 파라미터 조정은 3~4%p 개선에 머물렀고, S4처럼 전략 축을 바꾼 변경이 크게 뛰었다. S6는 봇끼리 강했지만 사람 상대 2판(2026-06-15 user 31:22, 2026-06-16 user 32:20)에서 졌다. 그래서 S7부터는 **사람 vs 봇 로그 분석을 주 지표**로 삼는다.

봇끼리 대량전은 폐기할 지표가 아니라 **회귀 체온계**로만 쓴다. 새 변경이 명백히 망가졌는지 확인하는 자동 안전망으로 유지하고, 강함 평가는 사람 로그와 특정 미스플레이 재현 테스트가 맡는다. 현재 기준선은 `HeadlessBotGameTest.S6vsS7SeedReport` 대표본(`-DbotSeedCount=250`, 선후공 2판 = 500판): S7 291승 / S6 209승, S7 승률 58.2%, 평균 코인 S7 28.81 vs S6 27.30, 무효 환금 0건(2026-06-17 실행).

## 3. S6/S7 핵심 교훈

과거 로그별 장문 분석은 코드와 테스트에 반영된 내용만 남기고, 앞으로도 유지해야 할 교훈 위주로 정리한다.

### 3.1 S6에서 얻은 것

- `predictedChooserValue`: 공개 코인만 보던 선택자 예측을 즉시 코인 + 상대 보관 시너지 + 뒷면 기대값으로 확장했다.
- 상시 maximin: 종반이 아니어도 최악의 경우를 일부 반영해 무방비 분할을 줄였다.
- 뒷면 저주 떠넘김: 보이는 저주는 상대가 피하므로, 뒷면 저주일 때만 떠넘김 보너스를 준다.
- 다중 세트 보류: 작은 세트를 즉시 환금하지 않고 성장 기대값이 큰 세트를 여러 개 보류한다.
- `PlayLogWriter`: 사람 vs 봇 분석의 기반을 만들었다.

### 3.2 S7에서 고친 것

- **와일드 헌납 방지:** `hiddenEv`로 얇은 뒷면 묶음을 의심하고, `WILD_GRAB`으로 보이는/숨은 와일드의 확보·견제 가치를 크게 본다.
- **분할자 와일드 보상 분할:** 와일드는 숨겨도 숙련자가 읽을 수 있으므로, `predictedChooserValue`는 뒷면 와일드도 선택자가 가져간다고 예측한다. 봇은 헛된 숨김보다 와일드를 내주는 대신 남는 묶음을 알차게 짜야 한다.
- **이유 없는 1:4 방지:** 선택자 예측은 뒷면 의심만이 아니라 카드 물량(`CARD_MATERIAL`)을 본다. 4장 묶음은 대체로 1장+뒷면보다 매력적이다.
- **저주 부채:** `CURSE_LIABILITY`로 저주를 음의 가치로 보고, S7 환금 튜닝에서 저주를 보류 공간 계산에서 제외한다.
- **저주 처분 타이밍:** 무료 처분 보너스는 한도 초과일 때만 켜고, 바이퍼는 한도 초과 구제용으로 아낀다.
- **큰 세트 양성:** 같은색 연속 보류 임계를 낮추고(`0.5`), 성장 가중을 더하며, 보류 방이 부족하면 잡카드를 비운다.
- **종반 선택 마진:** 30코인을 먼저 찍어도 같은 라운드에 상대가 더 많이 벌면 진다. 임계 통과 묶음끼리는 `bestCashCoin(내 보관+묶음) - bestCashCoin(상대 보관+상대 묶음)` 마진으로 고른다.

### 3.3 아직 미검증/취약한 S7 포인트

- 봇이 **분할자로서 와일드**를 쥔 상황의 보상 분할은 로그 검증이 부족하다.
- 종반 실현 마진은 `bestCashCoin` 근사라서 도우미 보너스, 리더 효과, 다중 환금, 알파 즉승을 완전히 반영하지 않는다.
- 한도 5에서 loose 카드가 전부 페어/런 재료면 4·5장 보류가 여전히 막힐 수 있다.
- 사람의 장기 빌드(예: 다라운드에 걸친 5장 런 폭발)를 누적 추적하는 모델이 없다.
- 2v2에서는 `decideTeamDistribution` 기본 구현이 카드 수 균형 분배라 전술적으로 약하다.

## 4. 로드맵

> 주 지표는 사람 vs 봇 로그 분석이다. 봇 vs 봇 대량전은 회귀 안전망, 특정 미스플레이 테스트는 고정 안전핀으로 쓴다.

### 4.1 [다음 우선] 분석 가능한 로그/테스트 인프라 강화

S8로 넘어가기 전에, 봇이 왜 그 수를 골랐는지 로그에서 바로 보여야 한다.

작업:

- `S7BotStrategy` 또는 공용 평가 레이어에 선택/분할 점수 breakdown을 남길 수 있는 디버그 훅 추가.
- 로그에 후보별 점수 항목 기록: `coin`, `synergy`, `hiddenEv`, `wild`, `curse`, `deny`, `secureMargin`, `chosen`.
- 사람 플레이 로그 외에 특정 상황을 고정하는 테스트 추가:
  - 2026-06-17 `001705 R9`형 종반 선택: 임계 통과 묶음 둘 중 실현 마진 큰 묶음 선택.
  - 와일드 분할자 상황: 와일드를 숨기는 데 집착하지 않고 보상 분할 선택.
  - 한도 5에서 5장 런 보류/완성.
  - 바이퍼를 한도 여유 때 낭비하지 않음.
  - 저주 무료 처분 보너스가 한도 초과 때만 켜짐.

구현 위치:

- `PlayLogWriter`
- `S7BotStrategy` 또는 새 평가 결과 record
- `S7BotStrategyTest`
- `HeadlessBotGameTest`

완료 기준:

- 패배 로그를 보고 “가중치가 틀렸는지, 평가 항목이 빠졌는지”를 후보 점수표로 판단할 수 있다.
- S7의 알려진 미스플레이가 단위 테스트로 재현되고 고정된다.

### 4.2 [S8 핵심] 공개 정보 메모리/카드 카운팅

현재 봇은 공개된 `opponentHoldings`와 `discardPile`를 순간 평가에 쓰지만, 장기 상태 모델은 없다. 강한 봇은 공개된 카드 흐름을 계속 기억해야 한다.

작업:

- `BotMemory` 또는 `PublicCardTracker` 추가.
- 카드 위치 추적: 내 보관, 상대 보관, 버림 더미, 공개 후 덱 재혼합, 미지.
- 특정 카드의 생존/획득 가능성 평가:
  - 색런을 완성할 카드가 버림 더미에 있는가?
  - 상대 보관에 있어 견제해야 하는가?
  - 이미 소모되어 성장 기대가 낮아졌는가?
  - 와일드가 버림 더미에 있어 `JUNK_DEALER` 가치가 폭증하는가?

구현 위치:

- `BotPlayer`가 상태 있는 전략을 보유하므로 전략 객체 내부에 메모리를 둘 수 있다.
- 필요하면 `GameListener` 이벤트를 직접 받는 대신, 현재 각 결정 컨텍스트와 `CashInContext.discardPile/opponentHoldings`로 매번 재구성하는 무상태 버전부터 시작한다.

완료 기준:

- `CashInPlanOptimizer.expectedStepTurns`가 단순 총량 근사보다 정확한 공개 카드 위치 정보를 사용한다.
- 상대가 들고 있는 완성 직전 색런/같은숫자 세트를 분할·선택에서 더 강하게 견제한다.

### 4.3 [S8 핵심] 상대 선택 확률 모델

S7의 `predictedChooserValue`는 “합리적인 한 명”을 가정한다. 사람 상대 강함을 올리려면 상대가 실제로 어떤 선택을 해왔는지 반영해야 한다.

작업:

- `OpponentModel` 추가.
- 선택 로그로 성향 추정:
  - 공개 코인 중시
  - 카드 장수/물량 중시
  - 뒷면 의심 강도
  - 와일드 집착
  - 저주 회피
  - 상대 빌드 견제 성향
- 분할 평가는 `chooserTakesA` 단일 boolean 대신 `P(takesA)`로 계산한다.
- 사람 로그가 충분치 않을 때는 S7 기본값을 prior로 사용한다.

구현 위치:

- `S7BotStrategy`를 직접 키우기보다 `S8BotStrategy`에서 `OpponentModel`을 주입/소유.
- `scoreSplit`의 기대값 계산부.

완료 기준:

- 같은 분할 후보라도 상대 성향에 따라 점수가 달라진다.
- 뒷면을 잘 읽는 사람에게는 와일드 숨김을 덜 시도하고, 공개 코인에 약한 사람에게는 미끼 분할을 더 활용한다.

### 4.4 [S8 핵심] 실제 환금 결과 기반 종반 평가

종반에는 잠재력보다 “이번 환금 단계 종료 후 팀 코인 차이”가 전부다. 현재 `bestCashCoin` 근사는 빠르지만 정식 룰 보너스를 빠뜨린다.

작업:

- `CashInPlanOptimizer`에 “행동 목록”뿐 아니라 예상 코인/즉승 여부를 반환하는 평가 API 추가.
- 종반 `scoreChoice`/`scoreSplit`에서 다음을 반영:
  - 도우미 보너스(`CUCKOO`, `LEO`, `LUCKY`, `ALPHA`)
  - 리더 효과(`CHUCK`, `JOE`, `WISE`, `FLANKY`)
  - 다중 환금
  - 저주 처분 비용/회피
  - 상대가 받은 묶음으로 즉시 낼 수 있는 점수
- `SECURE_MARGIN_UNIT`의 입력을 `bestCashCoin`에서 `projectedCashResult`로 교체한다.

구현 위치:

- `CashInPlanOptimizer`
- 새 record 예: `CashInProjection(actions, coinsGained, instantWin, remainingCards)`
- `S7BotStrategy` 또는 `S8BotStrategy`

완료 기준:

- 종반 선택/분할이 세트 코인이 아니라 환금 단계 후 실제 팀 코인 마진을 기준으로 결정된다.
- 알파 즉승, 럭키 +7, 죠 +1 같은 판 뒤집기 요소를 놓치지 않는다.

### 4.5 [중요] 2v2 팀 분배 전략

현재 기본 `decideTeamDistribution`은 보유 장수가 적은 멤버에게 균등 분배한다. 2v2 정식 룰에서는 큰 약점이다.

작업:

- `TeamDistribution` 평가 함수 추가.
- 분배 기준:
  - 같은 세트를 완성/성장시킬 카드는 한 명에게 몰아준다.
  - 저주는 바이퍼 보유자 또는 매칭 처분 가능성이 높은 쪽에 준다.
  - 숫자 1은 `ALPHA` 보유자에게 몰아준다.
  - 같은색 5장 가능성은 `LUCKY`, 같은숫자 3장 이상은 `LEO`, 같은색 3장 이상은 `CUCKOO`와 연결한다.
  - 리더/멤버 보유 한도 차이와 `FLANKY` 보너스를 반영한다.

구현 위치:

- `BotStrategy.decideTeamDistribution` 기본 구현은 유지.
- `S8BotStrategy`에서 override.
- 필요하면 `TeamDistributionOptimizer` 추가.

완료 기준:

- 2v2에서 카드 수 균형보다 세트 완성/도우미 활용/한도 관리를 우선한다.
- 분배 후 각 멤버의 환금 가능성이 로그에 설명 가능하다.

### 4.6 [중요] 도우미 드래프트 상황화

현재 `helperDraftScore`는 정적 랭킹이다. 하지만 도우미 가치는 인원수, 리더 효과, 승리 코인, 난이도, 팀 구성에 따라 달라진다.

작업:

- 필요하면 `decideHelpers` 입력을 `HelperDraftContext`로 확장한다.
- 정적 순위에 상황 보정 추가:
  - 1v1 한도 5/6이면 `TUSKER`, `DOUG`, `VIPER` 가치 상승.
  - `LUCKY`는 같은색 런을 적극 보류하는 전략과 결합될 때 가치 상승.
  - `ALPHA`는 숫자1 추적/분배 전략이 있을 때 가치 상승.
  - `CROC_BROTHERS`는 공개된 사용 도우미와 시너지 평가 필요.

구현 위치:

- `BotStrategy.decideHelpers`
- `BotKind`/게임 준비 흐름의 컨텍스트 전달 여부 검토

완료 기준:

- 같은 도우미 후보라도 룰/팀/전략 환경에 따라 선택이 달라진다.

### 4.7 [그다음] 난이도별 봇

강한 봇을 먼저 만든 뒤, 약한 봇은 강한 결정 위에 노이즈를 얹어 만든다.

작업:

- `BotKind`에 `EASY`/`NORMAL`/`HARD` 추가.
- `HARD`: 최신 전략 풀파워.
- `NORMAL`: 일부 가중치 완화, 상대 모델/메모리 일부 비활성.
- `EASY`: 후보 상위권 중 확률 선택, 가끔 공개 코인만 보는 선택으로 약화.

완료 기준:

- 같은 코드 경로를 공유하면서도 체감 난이도가 구분된다.

### 4.8 [장기] 시뮬레이션/lookahead

상태 추적과 상대 모델이 생긴 뒤에야 의미가 있다.

작업:

- 공개 정보 기반으로 남은 덱 분포를 샘플링한다.
- 분할/선택/환금 한두 라운드 앞을 rollout한다.
- 상대 응답은 `OpponentModel`의 선택 확률을 사용한다.

완료 기준:

- 단기 점수상 손해지만 다음 라운드 대형 세트/상대 차단으로 이기는 수를 찾는다.
- 계산 시간이 플레이 템포를 해치지 않는다.

---

## 5. 검증 원칙

- 사람 vs 봇 로그가 강함의 주 지표다.
- 봇 vs 봇 승률은 회귀 체온계로만 쓴다. `HeadlessBotGameTest.S6vsS7SeedReport`처럼 같은 seed 표본에서 선후공을 바꿔 돌리고, 새 전략이 S7 대비 터무니없이 무너지면 되돌아본다.
- 중요한 사람 로그 미스플레이는 단위 테스트로 고정한다.
- 무효 행동은 0건이어야 한다. `HeadlessBotGameTest`와 액션 검증으로 잡는다.
- 로그에는 최종 행동뿐 아니라 후보와 점수 근거를 남겨야 한다.

---

## 6. 설계 원칙

- 새 전략은 `BotStrategy` 인터페이스를 유지한다. 단, 드래프트/메모리 등 컨텍스트가 부족한 경우에는 새 context record 추가를 검토한다.
- 게임 엔진 규칙 로직과 봇 평가 로직을 분리한다(`SetEvaluator`/`CashInEvaluator`/`CashInPlanOptimizer`).
- 봇 전용 추론 상태는 네트워크/UI 미러 DTO를 오염시키지 않는다.
- 봇에 넘기는 정보는 규칙상 공개된 것만 사용한다. 상대 도우미·손패 같은 비공개 원본은 직접 참조하지 않고, 공개 행동 기반의 확률 추정만 허용한다.
- 강한 봇을 먼저 만들고, 난이도 하향은 래퍼/노이즈/기능 제한으로 만든다.
